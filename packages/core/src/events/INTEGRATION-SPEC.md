# Bloom Journal — Events Module: Integration Spec

The contract for the precomputed world-events system. Pair this with
`CLAUDE-CODE-PROMPT.md`. All code referenced here is provided and pre-tested.

## 1. What this is

A static, **global** dataset of special days plus a thin runtime layer. The build step
computes everything once; the app looks events up by date and (optionally) enriches them
with the user's location. No runtime network calls, no per-user data in the shared file.

## 2. File layout

```
src/events/
  types.ts             # shared types — the runtime contract
  precompute-events.ts # BUILD-ONLY (Node). Generates events.json. Never imported at runtime.
  comets.seed.json     # curated comet list (hand-maintained, refreshed ~2×/year)
  events.json          # GENERATED by the script. Commit it. ~1,644 events for 2024–2050.
  enrich-location.ts   # runtime, optional location enrichment (pure TS, browser/RN-safe)
  events-runtime.ts    # runtime lookup: getEventsForDate + personal events (pure TS)
  scene-effects.ts     # SceneEffect token contract + helpers (pure TS, NO visuals)
  useDayEvents.ts      # React hook (the only React file; web + Expo)
  index.ts             # public barrel
  example-usage.tsx    # reference only; not imported
```

Purity boundary: everything except `useDayEvents.ts` is framework-agnostic and shared by
web + mobile. `precompute-events.ts` is the only Node file and runs only at build time.

## 3. Data model (`types.ts`)

```ts
type EventType =
  | "fullMoon" | "newMoon" | "supermoon" | "micromoon" | "blueMoon" | "blackMoon"
  | "lunarEclipse" | "solarEclipse"
  | "solstice" | "equinox" | "seasonTransition" | "crossQuarter" | "solarTerm"
  | "earthApsis" | "planetOpposition"
  | "meteorShower" | "comet"
  | "fridayThe13th" | "leapDay"
  | "birthday" | "appAnniversary";        // runtime-only, never in events.json

type Rarity = "common" | "uncommon" | "rare" | "epic";

interface WorldEvent {
  id: string; type: EventType; date: string /* YYYY-MM-DD */; instantUTC?: string;
  title: string; subtitle?: string; rarity: Rarity;
  activeRange?: { start: string; end: string };  // showers
  meta?: Record<string, unknown>;
}
```

`events.json` shape: `{ generatedAt, range, scope: "global", byDate: Record<YYYY-MM-DD, WorldEvent[]> }`.

## 4. Public API (`index.ts`)

```ts
// Runtime lookup
getEventsForDate(date: Date | string, user: EventsUserContext, opts?: DayEventsOptions): WorldEvent[]
personalEvents(isoDate: string, user: EventsUserContext): WorldEvent[]

interface DayEventsOptions {
  coords?: Coords | null;            // ONLY when location granted
  timeZoneOffsetMinutes?: number;    // bucket instant-events by user's local day
}

// Location enrichment (called for you inside getEventsForDate; exported for direct use)
enrichWithLocation(events: WorldEvent[], coords?: Coords | null): WorldEvent[]

// Scene contract (NO visuals — you implement what tokens render)
SCENE_EFFECT: Partial<Record<EventType, SceneEffect>>
primaryEvent(events: WorldEvent[]): WorldEvent | null          // highest-rarity event
sceneEffectsForDay(events: WorldEvent[]): ResolvedEffect[]     // sorted, most-notable first

// React (web + Expo)
useDayEvents(date, user, opts?): WorldEvent[]
```

`EventsUserContext = { birthday: string; appInstallDate: string }` — supplied by the app
from its own profile/store. `Coords = { latitude; longitude; elevationM? }`.

## 5. Location handling (the important part)

- The shared `events.json` is **global** — zero coordinates. It ships identically to
  every user.
- Location is **additive and optional.** With no `coords`, every event still appears; the
  only effect is that season labels resolve to the Northern default (a global default,
  not any specific place).
- With `coords` (only when the user grants permission, e.g. via your existing
  geolocation/weather flow), `enrichWithLocation` adds:
  - **Solar-eclipse local visibility** → `meta.local = { visible, kind, altitudeDeg }`,
    `visible` only when the Sun is above the horizon. Lunar eclipses need no check.
  - **Hemisphere-correct season name** → flips "Spring"↔"Autumn" etc. when latitude < 0.
- Never hardcode a location. The build step takes no observer.

## 6. Build & refresh

- `npm run precompute:events` → writes `events.json` next to the script. Commit the
  output; it's deterministic and small.
- **Solar terms** (24/yr, very on-theme East-Asian seasonal markers) are off by default;
  set `includeSolarTerms: true` in the script's `main()` to include them.
- **Refresh cadence:** comets ~twice a year from BAA / aerith.net (the only routine
  maintenance); meteor λ☉ optionally yearly from the IMO PDF; everything else never.
- Extend coverage by bumping `END_YEAR` in `main()`.

## 7. Web + mobile

Both import the same pure core; `useDayEvents` is identical (React hooks work in RN).
`astronomy-engine` is pure JS with no native deps (Hermes-safe); Metro and Vite both
import JSON natively. Place the module wherever the repo already shares code; don't
restructure the monorepo.

## 8. Edge cases & notes

- **Timezones.** `events.json` dates are UTC. Pass `timeZoneOffsetMinutes` (e.g.
  `-new Date().getTimezoneOffset()`) so an instant-based event near UTC midnight (a late
  full moon, an eclipse) buckets onto the user's correct local journal day. Pure-calendar
  events (Friday 13th, leap day, season transition, comet) are tz-independent and keep
  their stored date.
- **Feb 29 birthdays/anniversaries** are observed on Feb 28 in non-leap years (switch to
  Mar 1 in `matchesAnnual` if preferred).
- **Comet magnitudes are predictions** — phrase tentatively in UI; keep `meta.note`.
- **Performance.** Lookups are O(1) by date; shower windows and per-tz indexes are built
  once and cached. `enrichWithLocation` only runs astronomy math for solar-eclipse days
  when coords are present, so per-day cost is negligible.

## 9. Quick verification (what "working" looks like)

```ts
const user = { birthday: "1995-08-12", appInstallDate: "2024-12-14" };

getEventsForDate("2026-08-12", user)
// → Perseids peak, New Moon, Solar Eclipse, Your Birthday

getEventsForDate("2026-08-12", user, { coords: { latitude: 15.49, longitude: 73.82 } })
//   solar eclipse meta.local → { visible: false }   (not visible from Goa)

primaryEvent(getEventsForDate("2026-08-12", user))?.title   // → "Solar Eclipse" (epic)
```
